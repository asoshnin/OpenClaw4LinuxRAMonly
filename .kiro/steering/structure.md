# Project Structure

## Multi-Project Architecture

OpenClaw supports parallel, isolated project contexts via the **Silo Pattern**.

| Layer | Location | Purpose |
|---|---|---|
| **Global Hub** | `agentic_factory/` | Core skills, Global DB (`factory.db`), shared agents |
| **Project Silo** | Any dir containing `.factory_anchor` | Isolated `project.db`, docs, memory per project |

**Root Discovery Invariant:** The Source of Truth is the **Project Root**, dynamically
discovered at runtime by searching upward from the current working directory for a
`.factory_anchor` file. Absolute hardcoded paths are **never** used. If no anchor is
found, the system falls back to `OPENCLAW_WORKSPACE` env var, then the default workspace.

**Logical Path Resolution:** All paths are computed at runtime relative to the discovered
root via `config.find_project_root()`. Physical symlinks are **deprecated** in favour of
this logical resolution model.

## Key Patterns

- **CLI tools** use `argparse` subcommands. Entry point is always `if __name__ == "__main__"`.
- **DB path** is always passed as a positional CLI argument вЂ” never hardcoded.
- **Project root** is always resolved via `config.find_project_root()`, never as a literal string.
- **Sprint status** lives in the `tasks` and `sprints` tables of `factory.db` вЂ” never in flat files.
- **REGISTRY.md** is regenerated by running: `python3 librarian_ctl.py refresh-registry <db_path> <registry_path>`
